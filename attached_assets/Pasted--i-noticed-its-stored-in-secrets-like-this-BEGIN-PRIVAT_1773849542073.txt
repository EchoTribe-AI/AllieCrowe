The Key Problem — Newlines in Secrets
Environment variables stored as secrets are single-line strings. A PEM private key needs literal newline characters (\n) between its lines, but when you paste it into a secrets manager, those become the two-character escape sequence \n (backslash + n), not actual newlines.

The Fix — Line 19
raw_key = private_key_pem or os.environ.get("WALMART_API_PRIVATE_KEY", "")
self.private_key_pem = (raw_key or "").replace("\\n", "\n")
.replace("\\n", "\n") converts the literal \n escape sequences back into real newline characters, so the PEM block looks like:

-----BEGIN PRIVATE KEY-----
xxxxx=
-----END PRIVATE KEY-----
instead of the flat single-line string from the secret.

How It's Used for Signing
Every API request to Walmart requires RSA-SHA256 authentication. Here's the full flow:

String to sign is assembled:

consumerId\n
timestamp_milliseconds\n
key_version\n
The PEM key is loaded using Python's cryptography library:

private_key = serialization.load_pem_private_key(
    self.private_key_pem.encode("utf-8"), password=None
)
Signed with PKCS1v15 + SHA256, then base64-encoded:

sig_bytes = private_key.sign(string_to_sign, padding.PKCS1v15(), hashes.SHA256())
signature = base64.b64encode(sig_bytes).decode("utf-8")
Sent as request headers on every call:

WM_CONSUMER.ID          → your consumer ID (public key)
WM_CONSUMER.INTIMESTAMP → timestamp used in the signature
WM_SEC.KEY_VERSION      → "1"
WM_SEC.AUTH_SIGNATURE   → the base64 signature
What's Stored in Secrets
Secret Name	Value
WALMART_API_PRIVATE_KEY	The full PEM key with \n as escape sequences
WALMART_API_PUBLIC_KEY	Your consumer ID (UUID-style string)
WALMART_PUBLISHER_ID	Your publisher ID for affiliate tracking
The \n → real newline replacement is the critical detail — without it, load_pem_private_key() would throw an error because it can't parse a malformed single-line PEM block.
